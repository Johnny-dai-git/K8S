Part 1 · The Big Picture

The Architecture: Control Plane and Data Plane

Six core components, one entry point.

A Kubernetes cluster has two layers. The control plane holds cluster state and runs the components that decide what should exist: API Server, etcd, Scheduler, and Controller Manager. The data plane is the set of worker nodes that run the containers, with kubelet and kube-proxy on each node. The two layers do not talk to each other directly. Every read, write, and watch event goes through the API Server. This is the single property that defines how the rest of the system behaves.

Six components. One single source of truth (etcd, fronted by API Server).
Nobody talks to anyone except the API Server.

1.1 The six components, in plain words

etcd
Source of truth

Stores every Kubernetes resource object (Pods, Services, Deployments, ConfigMaps, Secrets, ServiceAccounts…), each carrying its desired state (spec) and current state (status) on the same record. For a Pod: spec = image, CPU/memory requests, target node; status = phase (Running/Pending), assigned node, Pod IP.

API Server
Entry point + event hub

The cluster's unified entry point and event distribution hub. Every kubectl command, every controller read/write, every kubelet status update flows through here. It authenticates the caller, validates the change, persists it to etcd as a resource object, and broadcasts the change as a watch event. Crucially, it doesn't decide actions (that's the controller's job) and doesn't schedule Pods (Scheduler's job). It only records state and tells everyone what changed. Full lifecycle in §1.3.

Scheduler
Pod-to-node matchmaker

Watches the API Server for Pods with empty spec.nodeName, then runs a two-phase pipeline.
Filter: drop nodes that fail any constraint, resource fit (CPU / memory / Pod count), nodeSelector / nodeAffinity, taints & tolerations, topology spread, port and volume availability, node Ready & schedulable. The survivors are the feasible nodes.
Score: rank the feasible nodes with weighted plugins (LeastAllocated, BalancedAllocation, NodeAffinity, PodAffinity / PodAntiAffinity, TopologySpread, ImageLocality, …) and pick the highest total.
Writes the winning node into spec.nodeName. Doesn't touch the node itself; kubelet takes it from there.

Controller Manager
A bag of reconcilers

A collection of independent controllers, one per resource type: Deployment, ReplicaSet, Node, Endpoints, Job, ServiceAccount, and dozens more. Each subscribes to the API Server's watch stream for its own resources (never reading etcd directly), then runs a continuous reconcile loop: read the latest status, compare against the declared spec, and create/update/delete objects through the API Server to close the gap. Always watching, never one-shot. This is the engine that gives Kubernetes its self-healing behavior.

kubelet
Node agent

The node's execution agent. Watches the API Server for Pods bound to its own node, then runs a local reconcile loop: pull images and run containers via CRI (containerd / CRI-O), enforce CPU / memory limits via cgroups, mount volumes, run liveness / readiness / startup probes, and continuously report Pod.status plus node heartbeat back to the API Server. It executes; it doesn't decide what Pods exist (controller's job) or where they run (scheduler's job).

kube-proxy
Service plumber

One per node, the data-plane plumber for Service traffic. Watches Service and Endpoints objects, then programs the kernel (iptables or ipvs) so packets to a Service's ClusterIP (the cluster-internal stable virtual IP, routable only inside the cluster) get DNAT'd and load-balanced across the live backend Pod IPs. The mapping is dynamic: as Pods come and go, kube-proxy rewrites the rules in milliseconds. It is not in the data path itself; the kernel is. Mental model: stable internal IP (Service) → dynamic Pods, mapped by kube-proxy.

1.2 etcd, by example: every object's full state

etcd is described as "the cluster's database," but that label is too vague to work with. The exact contents matter.

etcd stores every Kubernetes resource object as a key-value pair, where:

  • The key is a hierarchical path identifying the object.
  • The value is the full serialized resource object, JSON or protobuf.

Concretely:

Key:    /registry/pods/default/nginx-abc

Value:  {
          "apiVersion": "v1",
          "kind": "Pod",
          "metadata": { "name": "nginx-abc", ... },
          "spec":     { "containers": [...], "nodeName": "node-2", ... },
          "status":   { "phase": "Running", "conditions": [...], ... }
        }

Two properties to note:

① Desired and current state are stored in the same object.

  • spec, the desired state. What the user or a controller declared.
  • status, the current state. What kubelet or another reporter observed.

etcd is the cluster's single source of truth because both halves live on the same record.

② etcd does not run logic. It stores objects and notifies watchers when objects change. The reconciliation logic, diffing spec against status and writing the next change, lives in controllers.

A complete reconciliation, end to end

Apply this manifest:

apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 3
  ...

What happens, step by step:

  1. API Server writes to etcd. A new Deployment object now lives at /registry/deployments/default/web with spec.replicas: 3 and status.replicas: 0.
  2. The Deployment controller is watching (through API Server). It receives the event: "new Deployment, status doesn't match spec."
  3. It runs its reconcile loop: 
    if current.replicas != desired.replicas:
    create (desired - current) Pods via API Server
  4. Three Pod objects appear in etcd, each with spec.nodeName == "". These are unbound Pods: created in storage but not yet assigned to any node.
  5. Scheduler is watching for unbound Pods. It picks nodes and writes spec.nodeName back.
  6. kubelet on each chosen node is watching for Pods bound to itself. It pulls images, mounts volumes, applies cgroup limits (CPU / memory), starts containers via CRI, runs liveness / readiness / startup probes, observes container lifecycle events (exit codes, OOM kills, restarts), and continuously updates status: Pending → ContainerCreating → Running → Ready. It also reports node heartbeat and condition (Ready / NotReady) back to the API Server.
  7. Once status matches spec, the reconcile loop sees current == desired and stops acting. But it never stops watching.

Why strong consistency is required

etcd uses Raft for strong consistency. This is required by the architecture, not optional. If two scheduler replicas read different views of node capacity, both could bind a Pod to a node that fits only one. With strong consistency and optimistic concurrency (each object carries a resourceVersion), the second writer's update is rejected and the writer retries on fresh data. Two controllers cannot act on stale state.

Why controllers do not read etcd directly

If etcd is the source of truth, why does every read and write go through the API Server? Three reasons:

  • AuthN/AuthZ. API Server enforces RBAC. etcd has no concept of users or permissions.
  • Validation and admission. Schema validation, defaulting, mutating and validating webhooks all run at the API Server. etcd stores bytes.
  • Watch fan-out. The API Server multiplexes one etcd watch into thousands of consumer watches. Without this layer, every controller would open a direct connection to etcd and overload it.
★
Interview-grade summary

Controllers do not read etcd directly; they watch the API Server and run reconcile loops that drive the system from current state toward desired state.

1.3 API Server: request lifecycle and controller integration

The card above listed four jobs: authenticate, validate, persist, broadcast. Each step has specific behavior worth examining.

What API Server does on every request

Any request, whether from kubectl apply, a controller update, or a kubelet status report, runs through five steps in order:

  • Authenticate. The API Server verifies caller identity using a TLS client certificate, bearer token, OIDC token, or (for in-cluster controllers) a ServiceAccount token mounted into the Pod. Unverified requests return 401 Unauthorized.
  • Authorize. RBAC checks whether this identity has permission to perform the requested verb (get, create, update, delete) on this resource type in this namespace. A controller running as default/web-controller trying to delete Pods in kube-system returns 403 Forbidden.
  • Admit. The object is checked against the OpenAPI schema, default values are filled in, and admission webhooks run. Webhooks include mutating (e.g. Istio's sidecar injector adds an envoy container) and validating (e.g. Pod Security Standards rejects privileged Pods). Any rejection stops the request before storage is touched.
  • Persist. The accepted object is serialized and written to etcd. etcd assigns a new resourceVersion (a monotonically increasing counter) used for optimistic concurrency.
  • Notify. The API Server reads etcd's change feed and pushes the change to every subscribed client as an ADDED / MODIFIED / DELETED event. One etcd watch fans out to thousands of consumer watches; etcd is not touched again for the broadcast.

The notify step is the basis of Kubernetes' event-driven model. Without it, controllers would need to poll for changes, generating constant load on etcd. With it, controllers stay idle until a relevant change arrives.

What "reconcile" means

Every controller runs the same inner loop, called the reconcile loop:

on every relevant event from API Server:
    desired = read spec   from API Server   ← what should be
    current = read status from API Server   ← what is

    if current == desired:
        do nothing                           ← system is converged
    else:
        action = create / update / delete a resource object
        send that write through API Server   ← API Server persists to etcd
                                                and broadcasts a new watch event

The point to fix in mind: a controller's "action" is not an imperative command. It is a CRUD write on a resource object in etcd, performed through the API Server. Controllers cannot start containers, send packets, or call other controllers. Their only output is a write that creates, updates, or deletes a resource object. Every other component reacts to that write through its own watch.

For example, the Deployment controller reads spec.replicas: 3 and status.replicas: 1, computes a gap of 2, and calls POST /api/v1/pods twice. Each POST goes through the full five-step lifecycle (auth → authz → admit → persist → notify). The persist step adds two new Pod objects to etcd; the notify event then triggers the scheduler's reconcile loop, which writes spec.nodeName back through the same API Server, which triggers kubelet, and so on. State propagates through the cluster one resource-object write at a time.

How API Server and controllers cooperate

The full flow has two writes to etcd: one for the user's spec, and a second for the resource object that the controller writes as its action. Both go through the API Server.

The controller's only output is a resource object write. There is no other channel: no commands, no callbacks, no direct execution. Every state change in the cluster is persisted to etcd as one of these two writes, and every component reacts only to the watch events that follow.

Why there are many controllers, and what this gives the cluster

The diagram above shows a single controller for clarity. In practice a cluster runs dozens of controllers in parallel, each watching one resource type and running its own reconcile loop. The stacked outline in the diagram is a reminder of this multiplicity. The single-responsibility design produces four architectural properties:

  • Single responsibility and decoupling. Each controller owns one resource and one reconciliation rule: Deployment Controller maintains replica count, Job Controller drives a Job to completion, Node Controller manages node Ready state and Pod eviction, Endpoints Controller keeps Endpoints in sync with Service selectors and Pod readiness. Controller A has no reference to Controller B. It only sees events on the resource types it watches.
  • Extensibility. Custom Resource Definitions (CRDs) and custom controllers extend the system without modifying the core. Argo CD adds a GitOps controller. Database operators (Postgres, Kafka, etcd-operator) add domain-specific reconcile loops. ML platforms add training-job controllers. Kubernetes is best understood as a controller framework rather than a fixed set of features.
  • Fault tolerance and crash recovery. Each controller is its own process running an idempotent watch → compare → reconcile loop. State lives in etcd, not in the controller. When a controller restarts, it issues a LIST for a full state snapshot and resumes WATCH from the latest resourceVersion. There is no central brain to fail. State drift, manual edits, partial outages are corrected on the next reconcile.
  • Auditability. Every change goes through one component. The API Server's audit log records who performed which action, when, and whether it was allowed.

1.4 The watch-and-reconcile pattern, end-to-end

With the request lifecycle and reconcile loop defined, the full sequence behind kubectl apply -f deployment.yaml is straightforward. No component invokes another. Each one watches the API Server for the resource types it owns: the scheduler watches Pods with empty nodeName, kubelet on node-3 watches Pods bound to node-3, the Deployment controller watches Deployments and ReplicaSets.

A single kubectl apply triggers this chain of independent reactions:

1.  kubectl  → API Server: create Deployment "web"
2.  API Server → etcd:       persist; emit watch event
3.  Deployment Controller (was watching):
     "I see a new Deployment, no matching ReplicaSet"
     → API Server: create ReplicaSet "web-abc"
4.  ReplicaSet Controller (was watching):
     "I see a ReplicaSet wanting 3 Pods, 0 exist"
     → API Server: create Pod × 3 (nodeName empty)
5.  Scheduler (was watching):
     "I see 3 unbound Pods"
     → API Server: bind Pod-1 → node-2, Pod-2 → node-1 ...
6.  kubelet on node-2 (was watching):
     "I see a Pod bound to me"
     → pull image, start pause, run probes
7.  kubelet → API Server: update Pod status to Running
8.  Endpoints Controller (was watching Services + Pods):
     "Service web-svc selects label app=web; Pod-1 has it"
     → update Endpoints object
9.  kube-proxy on every node (was watching Endpoints):
     "Endpoints changed; rewrite iptables"

No component issues a direct command in this flow. Each one observes a state change and acts on its own. This decoupling is why no single component is on the critical path. If the scheduler is stopped, existing Pods continue running and only new scheduling stops. When the scheduler restarts, it resumes from etcd's current state.

The architectural lesson of Part 1

The art of Kubernetes is decoupling. No component calls another. No component holds global state. Each one watches the API Server for resource changes, performs a single job, and writes the result back as a resource object. The only shared truth is etcd. This is what makes the system resilient (any component can crash and restart without losing state), extensible (adding a CRD plus a new controller is a complete extension mechanism), and operable at scale (every change is auditable through one log). Every other property of Kubernetes follows from this one design choice.

Part 2 · The Front Door

How external traffic reaches a workload

Four service types, one packet path.

The question this section answers: how does traffic from outside the cluster reach a Pod inside?

Pods receive IPs from a private CIDR that is routable only inside the cluster, and those IPs change every time a Pod is recreated. A Service provides a stable address in front of a changing set of Pods. The four Service types extend that address outward in stages, from cluster-internal only to a public HTTP entry point:

2.1 The four service types

ClusterIP
Virtual IP allocated from the Service CIDR, programmed into each node's kernel as iptables / ipvs DNAT rules by kube-proxy. Routable only inside the cluster.
In-cluster service discovery and east-west traffic. Default for all Services.
NodePort
Allocates a static L4 port (30000–32767) on every node. Traffic to node-IP:NodePort is forwarded to the ClusterIP. Layered on top of ClusterIP.
External L4 access without a cloud LB; bare-metal clusters; underlying transport for LoadBalancer.
LoadBalancer
Provisions an external L4 cloud load balancer (AWS NLB, GCP LB, Azure LB) via the cloud-controller-manager. The LB targets each healthy node's NodePort.
Single-Service external L4 exposure on managed clusters. Billed per LB.
Ingress
L7 (HTTP/HTTPS) routing layer. An Ingress Controller Pod (nginx, envoy, traefik) reads Ingress rules and proxies host/path-routed traffic, exposed via one shared LoadBalancer.
Multi-Service HTTP entry point with TLS termination, host/path routing, rate limiting, and auth.

The four types stack: each one builds on the previous. The remainder of this section walks through each in detail.

ClusterIP, the cluster-internal stable address

ClusterIP is the default Service type. The control plane allocates a virtual IP from the cluster's Service CIDR (for example, 10.96.0.0/12) and assigns it to the Service. The IP is routable only inside the cluster. No network interface anywhere binds it; it exists purely as the target of iptables or ipvs rules that kube-proxy programs on every node. Packets to the ClusterIP are DNAT'd to one of the live backend Pod IPs.

Each Service also gets a DNS name of the form name.namespace.svc.cluster.local, resolved by CoreDNS. In-cluster service-to-service calls (frontend → backend, app → cache, app → database) use this DNS name and never need to know Pod IPs. ClusterIP is not reachable from outside the cluster, which is why it is also the safe default.

NodePort, the same port on every node

NodePort builds on ClusterIP. In addition to allocating a ClusterIP, the control plane opens a static TCP/UDP port (default range 30000–32767) on every node in the cluster. Traffic sent to <any-node-IP>:<NodePort> is forwarded by kube-proxy to the Service's ClusterIP, then DNAT'd to a backend Pod.

NodePort is the simplest way to make a Service reachable from outside the cluster. It is common in bare-metal environments without a cloud load balancer, in dev or debug setups, and as the underlying layer that LoadBalancer Services depend on. The limitations are that ports are restricted to the high range, callers must know at least one node IP, there is no TLS termination, and there are no L7 features.

LoadBalancer, a cloud LB in front of NodePort

LoadBalancer builds on NodePort. The cloud controller manager calls the cloud provider's API (AWS NLB or ALB, GCP Load Balancer, Azure Load Balancer) to provision an external load balancer and assigns it a public IP or DNS name. The cloud LB forwards traffic to the NodePort on each healthy node, which then forwards to a backend Pod.

This is the standard way to expose a single Service to the internet on a managed cluster. The trade-off is cost: each LoadBalancer Service typically provisions its own cloud LB, and cloud LBs are billed per hour plus per byte. For a cluster with many externally-exposed Services, this becomes expensive. The next type addresses that.

Ingress, one entry point for many Services

Ingress is an L7 (HTTP/HTTPS) router that fronts many Services through a single external entry point. It has two parts: the Ingress resource (a Kubernetes object that declares routing rules: which host and path map to which Service), and the Ingress Controller (a Pod, typically running nginx, envoy, or traefik, that reads those rules and actually proxies requests). The Controller is exposed as a LoadBalancer Service, so all external HTTP traffic enters through one cloud LB.

How the Controller actually reaches a Pod. This is the core mechanism worth understanding. The Ingress Controller is itself a Pod running inside the cluster, which means it has full Pod-network reachability to every other Pod through the CNI plugin. It watches the Endpoints (or EndpointSlices) of every Service referenced by an Ingress rule, so it always holds the current list of healthy Pod IPs. When an HTTP request arrives, the Controller inspects the Host header and URL path, matches an Ingress rule, picks a Pod IP from the matching Service's Endpoints, and proxies the request directly to that Pod IP. The full data path is:

external client
   → cloud LB (LoadBalancer Service of the Ingress Controller)
   → NodePort on a node
   → Ingress Controller Pod
   → Pod IP of the target backend  (resolved from Endpoints, picked by Controller)

The Controller bypasses kube-proxy's Service DNAT for the final hop and proxies directly to a Pod IP. This direct Pod-IP proxy is what makes per-request L7 features possible: TLS termination, header rewrites, weighted traffic splits, sticky sessions, and retry policies all happen in the Controller before the request is forwarded.

Beyond routing, Ingress terminates TLS (typically paired with cert-manager for automatic certificate provisioning), and supports rate limiting, authentication, URL rewrites, and observability hooks. The economic argument is direct: one cloud LB plus one Controller can serve hundreds of HTTP backends. Ingress is the standard production pattern for exposing many HTTP services from a single cluster.

Because each type stacks on the one before it, the layering determines cost, failure modes, and latency for any external endpoint. A LoadBalancer Service has the failure modes of NodePort plus the cloud LB. An Ingress has the failure modes of LoadBalancer plus the controller Pod and its rule reload behavior. Choosing the right type is a balance between exposure surface, operational cost, and required L7 features.

2.2 The end-to-end packet path

The composition below is the canonical packet path used in production Kubernetes deployments, across all major clouds (AWS / GCP / Azure) and on-prem clusters. A request to https://api.example.com ultimately lands in a process inside a Pod on a node. The hops in between have converged into an industry-standard layering:

User ↓ DNS resolves to LB IP Cloud Load Balancer ↓ TCP/443 Ingress Controller (a Pod, e.g. nginx-ingress) ↓ TLS terminate, host / path routing Service (ClusterIP, e.g. 10.96.0.42) ↓ kube-proxy: iptables / ipvs DNAT Pod IP (e.g. 10.244.3.7) on Node-2 ↓ CNI: veth pair / overlay Container :8080 ← traffic finally arrives Every hop is a candidate root cause for a 503.

Each layer in this stack exists for a specific reason and is owned by a specific component:

  • Cloud LB: stable public IP / DNS, anycast, TLS offload (optional), DDoS protection. Provided by the cloud provider.
  • Ingress Controller: L7 routing (Host / path), TLS termination, rate limiting, auth, observability. Runs as a Pod in the cluster.
  • Service (ClusterIP): stable virtual IP and cluster-internal DNS for the backend. Programmed into kernel rules by kube-proxy.
  • kube-proxy: DNAT from ClusterIP to one of the live Pod IPs (iptables or ipvs). The final L4 load-balancing step.
  • CNI plugin: cross-node Pod connectivity (veth pair on the same node, overlay or BGP across nodes). Delivers the packet to the destination Pod's network namespace.

This composition is the de facto industry standard because it cleanly separates concerns: the cloud handles public-facing edge functions, the cluster handles application routing and load balancing, and the kernel handles the data path. The same separation also defines the diagnostic surface: any hop is a candidate root cause when something fails.

2.3 Services are kernel rules, not a proxy

A Service is not a process. There is no Service daemon and no Service Pod. The Service abstraction is implemented entirely as routing rules programmed into each node's Linux kernel by kube-proxy. The kube-proxy process itself only writes and updates these rules; it never sees a packet in the data path.

When a Service is created, the control plane allocates a virtual ClusterIP from the cluster's Service CIDR (for example, 10.96.0.42). This IP is not bound to any network interface. It exists only as a destination value that the kernel's netfilter (iptables) or IPVS subsystem matches against:

# What kube-proxy writes on each node (simplified):
-A KUBE-SERVICES -d 10.96.0.42/32 -p tcp --dport 80 \
  -j KUBE-SVC-XYZ

-A KUBE-SVC-XYZ -m statistic --mode random --probability 0.33 \
  -j KUBE-SEP-POD1   # 33% to Pod1
-A KUBE-SVC-XYZ -m statistic --mode random --probability 0.50 \
  -j KUBE-SEP-POD2   # 50% of remaining 67%
-A KUBE-SVC-XYZ -j KUBE-SEP-POD3                # fallthrough

-A KUBE-SEP-POD1 -j DNAT --to-destination 10.244.3.7:8080

The packet flow is straightforward: a packet to a ClusterIP enters the kernel, matches the KUBE-SERVICES rule, jumps to the per-Service chain, picks one backend by random probability, and is DNAT'd to that Pod IP. The packet never crosses the userspace boundary.

This kernel-only data path produces three direct consequences:

  • Performance. No context switches between kernel and userspace, no proxy buffer in the data path. Service load balancing runs at line rate.
  • Reliability. The data path does not depend on any userspace process. kube-proxy can crash and restart, and existing connections continue uninterrupted because the rules already programmed into the kernel keep working.
  • Scaling profile. The cost of a Service is the cost of a rule lookup in the kernel, not the cost of running a proxy. There is no per-Service memory or CPU footprint in userspace.

Two kernel implementations are available, with different scaling characteristics at large numbers of Services or endpoints:

iptables mode (default)
  • O(N) rule traversal per packet, N = endpoint count
  • Rule updates rewrite the entire chain (slow at large scale)
  • Degrades visibly past several thousand Services
  • Simple, stable, broadly deployed
ipvs mode
  • Hash-table lookup, O(1) per packet
  • Incremental rule updates, much faster reload
  • Multiple LB algorithms: rr, wrr, lc, sh, dh
  • Recommended for clusters with many Services or endpoints

2.4 Cross-node Pod connectivity: CNI

The core role of the Container Network Interface (CNI) plugin is one sentence: make every Pod able to reach every other Pod by IP, including across nodes. Without a CNI, Pods on different nodes have no path to each other; with a CNI, the Pod IP space behaves as a single flat network.

A CNI plugin owns three responsibilities on every node:

  • IP allocation. Assign each Pod an IP from the cluster's Pod CIDR (a non-overlapping subnet per node) when the Pod is created, and release it when the Pod is destroyed.
  • Per-Pod network configuration. Set up the network interface inside the Pod's network namespace, the host-side veth pair, and the routing/iptables/eBPF entries needed for the Pod to talk to the node, the Service network, and other Pods.
  • Cross-node connectivity. Ensure that a packet sent from a Pod on node A to a Pod IP on node B actually arrives, by either tunneling between nodes or programming routes into the underlying network.

The cross-node piece is implemented in one of two styles:

  • Overlay (Flannel VXLAN, Calico VXLAN/IPIP, Cilium VXLAN). Pod-to-Pod packets are encapsulated in VXLAN or IPIP between nodes. The underlying fabric only sees node-to-node traffic. Easy to deploy on any network, but each packet pays an encapsulation overhead.
  • Underlay / native routing (Calico BGP, Cilium native routing). Each node's Pod CIDR is announced into the physical network via BGP, so Pod traffic routes natively without encapsulation. Lower overhead and better observability, but requires cooperation from the network fabric.

The leading CNIs in production are Calico, Cilium, and Flannel. Cilium uses eBPF to implement the entire data path (Pod connectivity, Service load balancing, NetworkPolicy) in the kernel without iptables, which makes it the common choice for large or performance-sensitive clusters. Once a CNI is installed, the cluster has a working flat Pod network and the higher layers (Services, Ingress) can assume Pod-to-Pod reachability.

Part 3 · The Atom

A Pod is not a container. It is the unit of scheduling.

The abstraction layer above the container, and why it exists.

A Pod is not a wrapped container. A Pod is a group of containers that share Linux namespaces. Each Pod also contains a small pause container that the user does not declare.

3.1 The pause container

When kubelet starts a Pod on a node, the first container it creates is a small pause container (a few KB in size). The pause container has one job: hold the Pod's network namespace and IPC namespace open. Every other container in the Pod joins those namespaces.

All containers in a Pod share one IP, one port space, and one loopback interface. They reach each other over localhost. Traffic to other Pods leaves through eth0 and goes onto the cluster network.

This is why two containers in the same Pod cannot listen on the same port: they share one network namespace, and the port number is a property of that namespace. It also explains why the sidecar pattern (log agents, proxies, service mesh) is efficient: the sidecar shares loopback with the main container, so no extra network plumbing is required.

3.2 Why a Pod runs on exactly one node

A Pod cannot be split across nodes. It runs on exactly one node, always. The reason is direct: the Linux namespaces that hold the Pod together (network, IPC, optionally PID) are kernel objects that exist on a single machine. The kernel has no mechanism to share a namespace across machines. If a sidecar were placed on node A and the main container on node B, the two containers could not share the network namespace and the Pod would not function. The shared eth0, the shared port space, and the shared loopback all live in one kernel.

Kubernetes encodes this constraint in its scheduling model. The scheduler binds a Pod to one node by writing spec.nodeName, and kubelet on that node starts every container in the Pod inside the same namespaces. The Pod, not the container, is the unit of co-location.

The cost of this design is that Pods are largely immutable. The image of a running Pod can be changed, but most other fields (resources, networking, container list) cannot. Changes are made by creating a new Pod and deleting the old one. A Deployment rolling update is this operation, performed in a controlled sequence.

i
Design principle

A container is a packaging unit. A Pod is a scheduling and co-location unit that maps to one set of Linux namespaces on one machine. The Pod boundary exists precisely because shared kernel namespaces cannot cross machines.

Part 4 · The Brain

How Kubernetes schedules: Filter → Score → Bind

Three phases, and the difference between requests and limits.

Pod placement is not imperative. After kubectl apply, the Pod is written to etcd with spec.nodeName == "". The scheduler watches for these unbound Pods and runs a three-phase pipeline.

4.1 The three phases

# 1. Filter, which nodes could fit this Pod?
  for node in all_nodes:
      if node.allocatable < pod.requests:        drop
      if pod.tolerations don't cover node.taints: drop
      if pod.nodeSelector / affinity not met:     drop
  → candidate node set

# 2. Score, among candidates, which is best?
  for node in candidates:
      score = 0
      score += LeastRequestedPriority(node)   # idler nodes score higher
      score += ImageLocalityPriority(node)    # image already pulled? bonus
      score += InterPodAffinity(node)         # affinity satisfied? bonus
  → pick the highest

# 3. Bind, write back to etcd
  pod.spec.nodeName = chosen_node
  → kubelet sees "I have a new Pod" and gets to work

One important point about resources: the scheduler reads requests, not limits, and not actual usage.

resources:
  requests:
    cpu: 500m         # scheduler subtracts this from node.allocatable
    memory: 512Mi
  limits:
    cpu: 2000m        # enforced at runtime by cgroups, not by the scheduler
    memory: 1Gi

This produces a common scenario: a node shows 30% CPU usage but new Pods stay Pending. The cause is that the node is full by requests, even though existing Pods are not consuming what they reserved.

4.2 Affinity, taints, and tolerations

The scheduler honors three placement constraint mechanisms during the Filter phase, each one addressing a different question.

nodeSelector / nodeAffinity, select nodes by label

The simpler form is nodeSelector: a flat key-value match against node labels. The Pod runs only on nodes whose labels match all listed entries.

# Pod runs only on SSD nodes in zone us-east-1a
nodeSelector:
  disktype: ssd
  topology.kubernetes.io/zone: us-east-1a

nodeAffinity is the richer form. It supports set operators (In, NotIn, Exists, Gt, Lt) and a hard / soft distinction:

  • requiredDuringSchedulingIgnoredDuringExecution: a hard constraint. The Pod is not scheduled to nodes that fail the rule.
  • preferredDuringSchedulingIgnoredDuringExecution: a soft preference with a weight (1–100). Failing nodes still pass, but matching nodes score higher.
# Hard: must run on A100 or H100 GPUs. Soft: prefer us-east-1a.
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: nvidia.com/gpu.product
          operator: In
          values: [A100, H100]
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      preference:
        matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values: [us-east-1a]

Common use cases: GPU type selection, SSD vs HDD storage, zone preference for data locality, CPU architecture (amd64 vs arm64), and isolation by environment label (env=prod).

Taints and tolerations, node-driven repulsion

Taints are the inverse of affinity. A taint applied to a node prevents any Pod from being scheduled there unless the Pod declares a matching toleration. The model: the node repels everyone by default, except those who carry the right pass.

A taint has three fields (key, value, effect). The effect determines behavior:

  • NoSchedule: scheduler will not place new Pods on this node. Existing Pods remain.
  • PreferNoSchedule: soft version. Scheduler avoids this node when possible but does not refuse it.
  • NoExecute: refuses new Pods and evicts existing Pods that lack the matching toleration.
# Apply a taint to a node
kubectl taint nodes node-1 dedicated=gpu:NoSchedule
# A Pod that tolerates that taint and is therefore eligible
spec:
  tolerations:
  - key: dedicated
    operator: Equal
    value: gpu
    effect: NoSchedule

Common use cases:

  • Control-plane nodes are tainted with node-role.kubernetes.io/control-plane:NoSchedule so user workloads do not land on them.
  • GPU nodes are tainted with nvidia.com/gpu:NoSchedule. Only GPU-aware workloads tolerate it, which lets cluster-autoscaler scale GPU nodes down to zero when no GPU Pod is pending.
  • Dedicated nodes per tenant in multi-tenant clusters: each tenant's Pods carry the matching toleration; other tenants stay off.
  • Node draining: when a node is cordoned for maintenance, kubelet automatically adds node.kubernetes.io/unschedulable:NoSchedule. Existing Pods stay until evicted.
  • Failure detection: node.kubernetes.io/not-ready:NoExecute is added when a node loses heartbeat, evicting Pods after a tolerationSeconds window.

Pod affinity and anti-affinity, placement relative to other Pods

These rules place a new Pod near (affinity) or away from (anti-affinity) Pods matching a label selector, within a topology domain. The key concept is topologyKey, a node label that defines the size of the domain:

  • kubernetes.io/hostname: the domain is one node.
  • topology.kubernetes.io/zone: the domain is one availability zone.
  • topology.kubernetes.io/region: the domain is one region.
# Anti-affinity: spread replicas across nodes (no two on the same host)
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchLabels: { app: web }
      topologyKey: kubernetes.io/hostname
# Affinity: co-locate Redis Pods on the same node as the web app
affinity:
  podAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchLabels: { app: web }
      topologyKey: kubernetes.io/hostname

Common use cases:

  • HA across nodes: a Deployment of three replicas with anti-affinity on hostname guarantees no two replicas share a node. Losing a single node still leaves two healthy replicas.
  • Spread across zones: anti-affinity on topology.kubernetes.io/zone spreads replicas across AZs for disaster tolerance.
  • Co-located cache: pair a sidecar cache Deployment with the app Deployment using affinity on hostname so cache and app share a node, eliminating cross-node latency.
  • topologySpreadConstraints is the modern replacement for anti-affinity-based spreading. It expresses "spread evenly across this topology" directly, with a maxSkew tolerance.

4.3 GPU scheduling: fragmentation and pooling

CPU and memory can be subdivided into millicores and megabytes. The default GPU model in Kubernetes cannot. The NVIDIA Device Plugin exposes a node's GPUs as the integer resource nvidia.com/gpu, and a Pod requests whole cards:

resources:
  limits:
    nvidia.com/gpu: 1   # one whole GPU; cannot ask for 0.5

Whole-card allocation produces two problems: fragmentation when total free GPUs are scattered across nodes, and under-utilization when a workload uses only a fraction of a card.

Node-1: [GPU0 ✗] [GPU1 ✓] [GPU2 ✓] [GPU3 ✗] Node-2: [GPU0 ✓] [GPU1 ✗] [GPU2 ✓] [GPU3 ✗] Node-3: [GPU0 ✓] [GPU1 ✗] [GPU2 ✗] [GPU3 ✓] 6 free GPUs in the cluster. A 4-GPU training job arrives. Pending. Fragmentation: the total is enough, but not contiguous.

Reducing fragmentation

Three scheduling-side techniques compact whole-card allocation:

  • Bin packing: bias the scheduler toward nodes that already have Pods so fragments compact. Replace LeastRequestedPriority with MostRequestedPriority in the score function.
  • Descheduler: periodically evict scattered Pods so the scheduler can re-pack. Requires workloads that tolerate restarts.
  • Topology-aware scheduling: NVIDIA's GPU Operator exposes which GPUs share an NVLink domain. The scheduler keeps a multi-GPU job within one socket so collective communication does not cross slow PCIe links.

GPU pooling: making one card serve many Pods

Whole-card allocation often wastes capacity. A small inference Pod needs 2 GB of VRAM and 5% of compute, but reserves a 40 GB A100. Several technologies allow one physical GPU to serve multiple Pods. They are not built into Kubernetes core; they extend the device plugin or schedule layer.

  • Time-slicing (NVIDIA GPU Operator). The plugin advertises one physical GPU as N virtual GPUs. Multiple Pods are scheduled onto the same card and the GPU runs them in time-shared slots. There is no memory or fault isolation between tenants. Suitable for trusted workloads with bursty usage.
  • CUDA MPS (Multi-Process Service). NVIDIA's runtime feature that allows multiple processes to share one GPU concurrently rather than serially. Higher utilization than time-slicing but still no isolation.
  • MIG, Multi-Instance GPU (A100, H100, H200). Hardware partitions one physical GPU into up to 7 hardware-isolated instances, each with its own SMs and VRAM. The device plugin exposes these as separate resources (nvidia.com/mig-1g.5gb, nvidia.com/mig-2g.10gb, etc.). Strong isolation, suitable for multi-tenant inference. Profiles are static and reconfiguring requires draining the GPU.
  • vGPU (NVIDIA AI Enterprise / vCS). Driver-level virtualization that exposes a card as several vGPUs with configurable VRAM. Requires a paid NVIDIA license and supported datacenter GPU.
  • Open-source fractional schedulers (HAMi, Volcano vGPU, Aliyun GPU-share). Implement fractional GPU allocation in software by intercepting CUDA calls. Pods request nvidia.com/gpumem: 4Gi and nvidia.com/gpucores: 30. Useful for clusters without H100 / MIG support.

Multi-node training: one job across many Pods

The opposite scenario: a job needs more GPUs than any single node holds. A 70B parameter LLM training run needs 8 or 16 H100s, but each node has only one or two. The job must run as many Pods (one per GPU per node), coordinated as a single distributed job.

Three constraints make this non-trivial in vanilla Kubernetes:

  • Gang scheduling. All N Pods of a job must be scheduled together. If 7 of 8 land and the 8th stays Pending, the 7 hold GPUs idle. Default Kubernetes scheduling is per-Pod, so partial allocation can deadlock the cluster: each waiting job holds resources another waiting job needs.
  • Rank and master discovery. Each Pod needs its rank (0..N-1), the world size, and the address of the master Pod. Pod IPs are random and ephemeral, so the operator must inject this through environment variables or a headless Service.
  • Collective communication. Workers exchange gradients via tight collective operations (NCCL all-reduce). If one Pod is slow, all others wait. Step time is bounded by the slowest worker plus network round-trip.

The Kubernetes ecosystem solves this with a layer above Pods:

  • Kubeflow Training Operators (PyTorchJob, MPIJob, TFJob): one CRD spawns N Pods from one or more replica specs (Master, Worker), injects RANK, WORLD_SIZE, MASTER_ADDR, and integrates with gang scheduling.
  • Volcano: a CNCF batch scheduler with native gang-scheduling. Pods are grouped via PodGroup; the scheduler refuses to place any Pod in the group until all of them can fit. Used widely for AI training and HPC.
  • JobSet (kubernetes-sigs/jobset): a newer upstream API for coordinating groups of Jobs with multiple templates, replica counts, completion semantics, and gang scheduling. The intended successor to ad-hoc operators.
# PyTorchJob: 1 master + 7 workers, each on a node with 1 GPU
apiVersion: kubeflow.org/v1
kind: PyTorchJob
metadata: { name: llama-train }
spec:
  pytorchReplicaSpecs:
    Master:
      replicas: 1
      template:
        spec:
          containers:
          - { name: pt, image: torch:2.4, resources: { limits: { nvidia.com/gpu: 1 } } }
    Worker:
      replicas: 7
      template:
        spec:
          containers:
          - { name: pt, image: torch:2.4, resources: { limits: { nvidia.com/gpu: 1 } } }

Two additional concerns for multi-node training:

  • Network fabric. Cross-node gradient sync over TCP is too slow for large models. Production AI clusters use RDMA over Ethernet (RoCE) or InfiniBand, exposed through specialized CNIs (AWS EFA, GCP TCPX, Mellanox MACVLAN). GPUDirect RDMA lets one GPU read another's memory across nodes without staging through host RAM.
  • Topology-aware placement. Workers should land on nodes within the same network leaf (rack or fabric zone) to minimize all-reduce latency. Schedulers expose this through node labels like topology.kubernetes.io/zone and rack-level labels added by the cluster operator.

Dynamic Resource Allocation (DRA), the new primitive

The classic device plugin model is limited to integer counts of opaque devices. Kubernetes 1.26 introduced Dynamic Resource Allocation (DRA), which generalizes resource management. A Pod claims a ResourceClaim, and a vendor-provided driver decides which device satisfies it. DRA can describe partitioned GPUs, NVLink-connected GPU sets, network-attached accelerators, and other devices that the integer model cannot. It is becoming the foundation for GPU pooling in Kubernetes.

!
Production case

An LLM training cluster ran at 40% GPU utilization. Every researcher requested 2 GPUs for experiments, and the default scheduler (LeastRequested) spread Pods across nodes. The cluster appeared 60% empty, but the free cards were isolated singletons, unusable for multi-GPU jobs. Switching to MostRequested with bin packing raised utilization to 78%. Adding MIG profiles for inference Pods on the same cluster raised effective capacity by another 30% by serving small inference workloads on partitioned GPUs alongside large training jobs.

Part 5 · The Heartbeat

Probes: how Kubernetes determines container state

liveness, readiness, and startup, three probes for three independent questions.

Kubernetes does not have direct knowledge of application health, because health is a business-level concept. It relies on the application to expose an endpoint that can be polled. If the endpoint responds successfully, Kubernetes treats the container as alive. This polling check is a probe.

"Alive" has three distinct meanings, and Kubernetes provides one probe for each:

livenessIs the process still there?on failure → restart container
readinessIs it ready to serve traffic?on failure → drop from endpoints, don't restart
startupHas it finished starting?while pending, the other two don't fire

What each probe catches in practice:

  • liveness catches a process that is technically running but no longer making progress: a Java service in a deadlock, a Go service whose handler goroutines have leaked, a Node.js process stuck in a tight CPU loop. The process is alive, but the application is dead. The liveness probe fails repeatedly, kubelet restarts the container, and the new instance starts clean.
  • readiness gates traffic during transient unfit states without restarting. A service loading a 5 GB model into memory at startup. A service whose database connection has dropped and is reconnecting. A Pod going through graceful shutdown. In all of these cases the container should stay running but should not receive traffic. Readiness fails, the Service controller drops the Pod from Endpoints, kubelet does not kill the container.
  • startup prevents premature liveness checks during slow boot. A legacy Java app with a two-minute Spring Boot initialization. An ML inference server loading multi-gigabyte model weights. A migration step that runs once on first start. Without a startupProbe, an aggressive 30-second liveness timeout would kill the container before it finishes initializing. With one configured, liveness and readiness stay suppressed until startup passes.

A typical YAML for an ML inference Pod that uses all three:

startupProbe:
  httpGet: { path: /healthz, port: 8080 }
  periodSeconds: 10
  failureThreshold: 30        # up to 30 × 10s = 5 minutes for model load

readinessProbe:
  httpGet: { path: /ready, port: 8080 }   # model loaded AND DB connected
  periodSeconds: 5
  failureThreshold: 3

livenessProbe:
  httpGet: { path: /healthz, port: 8080 } # process is responsive
  periodSeconds: 30
  failureThreshold: 3

5.2 A common configuration mistake, and why it produces 503 storms

Many 503 incidents trace back to a Pod that defines only a liveness probe and no readiness probe. The configuration looks reasonable, but the resulting behavior is wrong.

# Anti-pattern: liveness only, no readiness
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 60
# No readiness defined

What happens, step by step:

  1. The Pod starts. The container's main process is running.
  2. No readiness probe is defined. Kubernetes defaults to "the Pod is ready as soon as its container is running" and sets the Pod condition Ready=true immediately.
  3. The Endpoints controller sees Ready=true and adds the Pod IP to the Service's Endpoints object.
  4. kube-proxy on every node sees the new endpoint and writes iptables (or ipvs) rules that route Service traffic to it.
  5. Meanwhile, the application is still initializing: warming caches, opening DB connections, loading models. The HTTP listener is not yet serving real requests, or returns 5xx for them.
  6. Incoming traffic hits the Pod and gets 503 (or a connection timeout). Every request during the warmup window fails.
  7. After 30 seconds the application finishes warming up and starts succeeding. The 503 storm during warmup was entirely avoidable.

The liveness probe does not prevent this. Liveness only triggers a restart when /health fails, and during warmup /health happens to return 200 (the process is alive, just busy starting up). Liveness has no effect on traffic routing. It cannot tell the Service to wait.

The fix is to add a readiness probe that gates "actually able to serve traffic." Use two distinct endpoints, one for "process is alive" (liveness) and one for "ready for requests" (readiness):

readinessProbe:
  httpGet:
    path: /ready              # business-level: caches warm, deps connected
    port: 8080
  periodSeconds: 5
  failureThreshold: 3
livenessProbe:
  httpGet:
    path: /health             # minimal: process is responsive
    port: 8080
  periodSeconds: 30
  failureThreshold: 3
  initialDelaySeconds: 60     # give the app time to start

What happens with this configuration:

  1. Container starts. Both probes begin polling.
  2. /health returns 200 immediately. Liveness passes. The container is not restarted.
  3. /ready returns 503 while caches are warming. Readiness fails three times in a row.
  4. The Endpoints controller sees Ready=false and does not add the Pod to the Service's Endpoints.
  5. kube-proxy never routes traffic to this Pod. No user-visible 503s.
  6. After 30 seconds, /ready starts returning 200. Readiness passes. The Endpoints controller adds the Pod. Traffic begins flowing, and the Pod is ready for it.

The general rule: liveness answers "is this container alive?" and readiness answers "should it receive traffic?" They are different questions, owned by different controllers (kubelet for liveness restart, Endpoints controller for traffic routing), and need different probes pointing at different application endpoints.

Of the two, readiness is the more important probe to configure correctly. Three reasons:

  • It is the only traffic gate Kubernetes has. Without a readiness probe, the Endpoints controller treats every Pod as ready the moment its container starts. Every warmup window, every transient dependency loss, every graceful-shutdown phase becomes a 503 storm. There is no other mechanism in the platform to keep traffic away from a not-yet-ready Pod.
  • It has no destructive failure mode. A misconfigured liveness probe kills healthy containers under transient stress and creates restart loops that propagate the failure. A misconfigured readiness probe only takes the Pod out of rotation; the container keeps running and can recover. Readiness fails safely; liveness fails dangerously. Many production setups intentionally omit liveness for this reason but always keep readiness.
  • It is the contract between the Pod and the rest of the cluster. Service load balancing, rolling updates, autoscaling, and PodDisruptionBudget all read the Ready condition that readiness sets. A Pod without a meaningful readiness probe is effectively invisible to these mechanisms.

The readiness endpoint must reflect what the application actually needs in order to serve: caches warm, downstream connections established, migrations completed, model weights loaded. If /ready returns 200 the moment the HTTP listener binds, it is no better than having no probe at all.

★
If you define only one probe, define readiness

Readiness is the only mechanism in Kubernetes that can hold traffic back from a Pod that cannot yet serve, and the only probe that fails safely. Every production Pod that serves traffic should have one, pointing at an endpoint that genuinely reflects readiness, not just "the process started."

5.3 CrashLoopBackOff and exponential backoff

When a container exits repeatedly, kubelet does not restart it immediately. It applies exponential backoff: 10s, 20s, 40s, 80s, capped at 5 minutes. This prevents a container that fails on startup from pinning the node at 100% CPU through image pulls and rapid restarts.

CrashLoopBackOff is not "currently crashing." It is "kubelet is waiting before the next restart attempt." To see the actual exit reason, run kubectl describe pod and read the Last State section.

★
Operational implication

Self-healing in Kubernetes is built on probes. Correctly configured probes make the cluster more stable over time. Incorrectly configured probes (liveness too aggressive, readiness too loose) cause Kubernetes to restart healthy Pods or drain usable replicas, producing self-inflicted incidents.

5.4 Three failure modes, four configuration principles

Probes are not optional configuration. They are the control signals Kubernetes uses to decide when to restart a container, when to send it traffic, and when to wait. A misconfigured probe turns the self-healing mechanism into a self-destructive one. Three failure modes account for most probe-related production incidents:

Mode A: liveness too aggressive → CrashLoopBackOff

The container is slow to start (JVM init, large cache load, ML model weights). The liveness probe begins checking too early or has too tight a threshold. The probe fails, kubelet restarts the container, the new instance is also slow to start, the probe fails again. The cluster has now turned a slow-but-healthy app into a permanent CrashLoopBackOff. Root cause: liveness has no concept of "still starting."

Mode B: readiness too permissive → 503 storm

The readiness endpoint returns 200 too early — before the cache is warm, before the database connection is established, before the model is loaded. The Endpoints controller adds the Pod to the Service, traffic flows in, every request fails. The Pod looks healthy by Kubernetes' definition, but the application is not. This is the §5.2 anti-pattern stated as a general failure mode: readiness must reflect real serving readiness, not "the listener bound."

Mode C: no startup probe → killed mid-startup

Long-starting applications (JVM apps with full Spring context, ML inference servers loading multi-gigabyte weights, apps with significant cold-cache warmup) take minutes to initialize. Without a startup probe, liveness begins checking after initialDelaySeconds and fails because the app has not yet bound its endpoints. kubelet restarts the container before it ever gets to serve. The container never finishes booting.

Four configuration principles

  1. Readiness must be strict. Point the readiness endpoint at real dependency checks: database reachability, downstream service health, cache state, model loaded. If any required dependency is down, readiness must fail. The cost of a strict readiness probe is one Pod out of rotation. The cost of a loose one is a 503 storm.
  2. Liveness must be conservative. Only detect unrecoverable states. Do not kill the container for transient slowness, downstream unavailability, or temporary network blips. A common safe pattern is to make liveness check only that the process is responsive (e.g., a basic /healthz that returns 200 if the HTTP server is alive), with a generous threshold. Many production setups omit liveness entirely.
  3. Slow-starting applications need a startup probe. Anything that takes more than 30 seconds to become serviceable should have one: JVM apps, ML inference, anything with cold-cache warmup. The startup probe suppresses liveness and readiness until startup completes, then hands off.
  4. Tune the parameters; defaults are usually wrong. Five fields control probe behavior:
    • initialDelaySeconds: wait this long after container start before the first check.
    • periodSeconds: time between checks.
    • timeoutSeconds: how long to wait for a single check to respond.
    • failureThreshold: consecutive failures required to count as failed (avoid single-flake reactions).
    • successThreshold (readiness only): consecutive successes required to mark ready again after a failure.
    The defaults are tuned for very simple apps. For anything with real startup time, dependencies, or load characteristics, every value should be set deliberately.

The summary worth carrying out of this section: probes are control signals. liveness controls restarts. readiness controls traffic. startup gates the other two. Configured well, they make the cluster heal itself. Configured wrong, they cause the cluster to attack itself, and the resulting incidents (CrashLoopBackOff loops, 503 storms, draining cascades) are among the most common root causes in production Kubernetes.

Part 6 · The Replicators

Three controllers for three workload shapes

Deployment, StatefulSet, and DaemonSet each solve a different deployment problem.

Deployment is the default reach for most workloads, but the three controllers solve structurally different problems. Picking the wrong one creates work later when stateful behavior, ordering, or per-node placement becomes required.

Deployment
  • Stateless replicas, interchangeable
  • Random Pod names
  • Horizontal scale, rolling update
  • Typical: web APIs, workers, gateways
StatefulSet
  • Ordered, stable identity
  • Predictable names (pod-0, pod-1, …)
  • Each Pod gets its own PVC
  • Typical: databases, Kafka, ZooKeeper, etcd
DaemonSet
  • Exactly one Pod per node
  • Count tracks nodes, not load
  • Common host access (logs, network, kernel)
  • Typical: log shippers, metrics agents, CNI, runtime security

Example 1, Deployment: a stateless web API

Scenario. An HTTP API (Python, Node, Go, …) that needs N interchangeable replicas behind a Service.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-api
spec:
  replicas: 3
  selector:
    matchLabels: { app: web-api }
  template:
    metadata:
      labels: { app: web-api }
    spec:
      containers:
      - name: web
        image: nginx:latest
        ports:
        - { containerPort: 80 }

Behavior.

  • Creates 3 Pods with random names (web-api-7c9d8b...).
  • If any Pod dies, the ReplicaSet controller creates a replacement automatically.
  • kubectl scale deployment web-api --replicas=10 brings the count to 10 in seconds.
  • A new image triggers a rolling update: new and old ReplicaSets coexist while traffic shifts gradually.

What this controller is good for.

  • Pods are fungible: replacing one with another costs nothing.
  • No fixed identity, no per-Pod state.
  • Standard fit: web frontends, REST/gRPC APIs, async workers, batch consumers.

Example 2, StatefulSet: a database cluster

Scenario. A MySQL cluster with two replicas, where each instance must keep its own data and identity across restarts.

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql                # headless Service for stable DNS
  replicas: 2
  selector:
    matchLabels: { app: mysql }
  template:
    metadata:
      labels: { app: mysql }
    spec:
      containers:
      - name: mysql
        image: mysql:8
        ports:
        - { containerPort: 3306 }
        volumeMounts:
        - { name: data, mountPath: /var/lib/mysql }
  volumeClaimTemplates:             # one PVC per Pod, generated automatically
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests: { storage: 10Gi }

Behavior.

  • Pods are named deterministically: mysql-0, mysql-1.
  • Each Pod has its own 10 Gi PVC, attached for the life of that named Pod.
  • Pods start in order (mysql-0 first, then mysql-1) and shut down in reverse order.
  • Deleting mysql-0 recreates it as mysql-0 with the same PVC reattached, so the database files survive.
  • Each Pod gets a stable DNS name: mysql-0.mysql.<namespace>.svc.cluster.local.

What this controller is good for.

  • Pods have identity (other replicas can address them by name).
  • Pods have persistent data (PVC is bound to the named Pod, not random).
  • Pods have ordered lifecycle (matters for cluster bootstrap).
  • Standard fit: MySQL / Postgres / MongoDB clusters, Kafka, ZooKeeper, etcd, Elasticsearch.

Example 3, DaemonSet: a per-node agent

Scenario. A log collector (fluentd, fluent-bit) that must run on every node to ship that node's container logs.

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels: { app: fluentd }
  template:
    metadata:
      labels: { app: fluentd }
    spec:
      containers:
      - name: fluentd
        image: fluentd:latest
        resources:
          limits: { memory: 200Mi }

Behavior.

  • Every node automatically runs exactly one fluentd Pod.
  • Adding a node to the cluster causes a new fluentd Pod to start on that node.
  • Draining or removing a node removes its Pod.
  • There is no replicas field. The count is determined by node count, plus any node selectors and tolerations.

What this controller is good for.

  • Pod count tracks nodes, not load.
  • Standard fit: log shippers, metrics exporters (DCGM, node-exporter), CNI agents, kube-proxy, runtime security agents (Falco, SentinelOne), GPU device plugins.
i
Real-world classification: monitoring stack

A common interview question: how do you deploy DCGM exporter (NVIDIA GPU metrics) and Grafana on Kubernetes? DCGM exporter is a DaemonSet because it must run on every GPU node to read that node's GPU metrics. Grafana is a Deployment because it is a centralized UI and query layer that does not need a copy per node. The full stack: DCGM exporter (DaemonSet) → Prometheus (Deployment, scrapes metrics) → Grafana (Deployment, visualizes). Picking the right controller for each tier matters: a DaemonSet for the centralized service would waste resources; a Deployment for the per-node collector would miss data on nodes without a collector Pod.

The one-line decision rule:

  • Deployment when you need N interchangeable replicas (web servers, workers, API gateways, Ingress controllers, application pods).
  • StatefulSet when you need N instances with stable identity and per-instance storage (databases, message brokers, distributed coordination services).
  • DaemonSet when you need one instance per node (log shippers, metrics collectors, CNI agents, runtime security, GPU drivers).

6.1 HPA: how target replicas are computed

The Horizontal Pod Autoscaler (HPA) is a Kubernetes controller that automatically scales a Deployment or StatefulSet's replicas field based on observed metrics. It samples metrics from the metrics API every 15 seconds, applies a formula, and writes the new replica count back through the API Server. Like every other controller, its only output is a write to a resource object.

desiredReplicas = ceil( currentReplicas × (currentMetric / targetMetric) )

Example: 10 replicas, CPU averaging 80%, target 50%
desired = ceil(10 × 80/50) = ceil(16) = 16   → scale up

Example: 10 replicas, CPU averaging 30%, target 50%
desired = ceil(10 × 30/50) = ceil(6)  = 6    → scale down

A minimal HPA targeting CPU utilization:

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: web-hpa }
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-api
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50

Three limitations to know

  1. Lag. Metric collection, the HPA decision window, and Pod startup together total 30 to 90 seconds. During a sudden traffic spike, the existing replicas absorb load alone for that window.
  2. Oscillation. When load hovers near the target, replica counts ping-pong (scale up, then scale down, repeat). HPA v2 added behavior.scaleDown.stabilizationWindowSeconds (default 300s) to dampen this.
  3. Wrong metric. CPU is not a useful signal for I/O-bound services or GPU services. A request blocked on a database returns no CPU pressure, and a GPU service with a saturated GPU shows low CPU. Use custom metrics (QPS, queue depth, p99 latency, GPU utilization) exposed through a custom-metrics adapter.

6.2 HPA, Scheduler, Cluster Autoscaler: demand vs supply

HPA is one half of an autoscaling system. It scales demand, the Pod count. It does not scale supply, the underlying node count. This separation is fundamental and produces a specific failure mode worth understanding.

DEMAND scaling SUPPLY scaling ───────────────────── ───────────────────── HPA: how many Pods? Cluster Autoscaler: how many nodes? ↓ writes spec.replicas ↓ provisions / removes nodes ReplicaSet creates Pods Cloud API adds VMs ↓ ↓ Scheduler tries to place new Node joins, kubelet starts ↓ if resources fit → Running if resources don't → Pending

The three components have different concerns:

  • HPA watches load (CPU, queue depth, GPU utilization, custom metrics) and decides how many Pods should exist. It is blind to whether the cluster has the capacity to run them.
  • Scheduler watches Pods with empty nodeName and decides which node each one runs on. If no node has the required resources, the Pod stays Pending.
  • Cluster Autoscaler (CA) watches for Pending Pods and decides whether to add or remove nodes. It calls the cloud provider's API (AWS Auto Scaling Group, GCP MIG, Azure VMSS, Karpenter) to provision or release VMs.

What goes wrong without Cluster Autoscaler

HPA can make a correct scaling decision and still fail to deliver the capacity. The classic case:

HPA: queue depth is high, scale from 4 → 10 Pods
   ↓
ReplicaSet creates 6 new Pods
   ↓
Scheduler: cluster has 4 GPUs total, all already used
   ↓
6 Pods stay Pending forever
   ↓
queue keeps growing, HPA keeps "deciding correctly," nothing changes

HPA cannot fix this on its own. It does not understand "the cluster ran out of GPUs." It just keeps writing higher replica counts. The fix is to run Cluster Autoscaler alongside HPA: when CA sees the Pending Pods, it provisions new GPU nodes, the scheduler then places the Pods, and the system actually scales.

The complete autoscaling stack for GPU inference

Combining all three components plus a custom metric source, the production-grade GPU inference autoscaler looks like:

[GPU Node] ↓ DCGM exporter (DaemonSet) reads NVIDIA driver metrics ↓ exposes /metrics Prometheus (scrapes every 15s) ↓ custom-metrics-apiserver (translates Prometheus → K8s metrics API) ↓ HPA (reads queue_depth or DCGM_FI_DEV_GPU_UTIL, scales Pod replicas) ↓ scale up: writes Deployment.spec.replicas ReplicaSet creates new Pods (Pending if no GPU available) ↓ Cluster Autoscaler sees Pending GPU Pods ↓ calls cloud API: add a new GPU node new GPU node joins the cluster ↓ Scheduler places the Pending Pods on the new node ↓ new Pods start serving

Two practical notes:

  • Provisioning latency. A cloud GPU node takes 60 to 300 seconds to come up: VM provisioning, kernel boot, container runtime init, driver install, kubelet join, CNI ready. Plan for it: keep a small pool of warm nodes (overprovision) for spike absorption, or use Karpenter for faster, more flexible provisioning.
  • Choice of HPA metric for GPU. GPU utilization and pending request queue depth both work, but queue depth is usually more stable. GPU utilization can spike to 100% during normal batched inference without indicating overload. Queue depth, by contrast, only grows when the system is genuinely behind.
★
The two scaling axes

HPA does workload scaling (more Pods). Cluster Autoscaler does infrastructure scaling (more nodes). The Scheduler is the bridge: it places Pods only when supply meets demand. A production autoscaling story always involves all three. A configuration with HPA alone will hit a Pending wall the first time the cluster runs out of resources.

Part 7 · The Architectures

Four production system designs

Compositions of the primitives covered in earlier sections.

7.1 High-throughput API (millions of QPS)

Target: a global API at p99 < 100 ms, peak 1M QPS.

Each layer absorbs a portion of the traffic before it reaches the next. CDN handles the majority of static reads. Cloud LB and Ingress handle TLS, edge routing, and rate limiting. Service distributes inside the cluster. API Pods are stateless and scaled horizontally by HPA. Redis absorbs hot reads so the database sees only true cache misses. Each tier is independently scalable, which is the property that makes million-QPS systems possible.

Key decisions:

  • Cache first. A 10% improvement in cache hit rate reduces DB load by approximately one order of magnitude. Redis must run in the same region as the API Pods.
  • HPA on a business metric. Use QPS or in-flight request count. CPU is a lagging signal.
  • Pod anti-affinity. Replicas of one Deployment should not be packed onto a single node. Losing one node would remove a large fraction of capacity.
  • PodDisruptionBudget. Define the minimum number of Pods that must remain available during rolling updates and node maintenance.

Why this architecture handles high concurrency. The throughput comes from one property repeated at every layer: statelessness + horizontal scaling. CDN, cloud LB, Ingress Controller, Service, and API Pods are all stateless or replicable. Adding more replicas at any layer adds capacity linearly. The only stateful components (Redis, database) have their own scaling strategies (Redis cluster sharding, DB read replicas). Ingress alone does not produce high QPS; the entire stack is designed so that no single component becomes a per-request bottleneck. Cache sits at every layer (CDN edge, Redis, DB query cache) so most requests never reach the slowest component.

7.2 LLM inference platform

Target: serve a 70B parameter model with dynamic batching and on-demand GPU scaling.

# Key spec for a GPU worker
spec:
  containers:
  - name: vllm-worker
    resources:
      limits:
        nvidia.com/gpu: 4              # 4 × A100
    env:
    - name: TENSOR_PARALLEL_SIZE
      value: "4"
  nodeSelector:
    accelerator: a100-80gb
  tolerations:
  - key: gpu                            # GPU nodes carry a taint
    operator: Exists

Three independent layers, glued by a queue. The API Gateway (cheap CPU Pods) accepts requests at line rate and enqueues them. GPU Workers (expensive, slow to start) pull at their own pace. The queue absorbs the speed mismatch. HPA watches queue depth as the primary signal because GPU utilization lags behind actual load.

The architectural property is decoupling:

  1. The API gateway only enqueues requests. Its scaling is inexpensive because it runs on CPU Pods.
  2. GPU workers have slow startup (pulling model weights, warmup). The queue absorbs traffic while new workers come online.
  3. HPA watches queue depth instead of GPU utilization. GPU utilization is a lagging signal: by the time it is pinned, latency has already increased.

Why decoupling is the right pattern for GPU inference. The producer (API gateway, microseconds per request) and the consumer (GPU worker, hundreds of milliseconds per inference, minutes to start) operate on incompatible timescales. A direct call from gateway to worker would block the gateway and time out under any traffic spike. The queue absorbs this speed mismatch: gateways write at their natural rate, workers read at theirs, and the queue grows or shrinks to track the difference. The queue also enables three properties that direct synchronous calls cannot provide: independent scaling (each tier scales on its own metric), request retention during scale-up (requests sit in the queue while new workers boot), and backpressure visibility (queue depth is a single, observable signal of system health). This is the same decoupling principle that streaming pipelines use, applied at the request level rather than at the event level.

7.3 Runtime security agent (DaemonSet)

This is the architecture used by Falco, Sysdig, SentinelOne, and similar tools. One agent per node listens to kernel events (syscalls via eBPF) and ships events to a central detection engine.

The DaemonSet agent reads kernel-level events through eBPF probes attached to syscalls, packets, and file operations. Events from every node converge into a central detection engine via gRPC or Kafka, where rules and ML detectors generate alerts, drive automated SOAR responses, and feed storage for forensics.

This must be a DaemonSet rather than a Deployment because the agent needs to read kernel events on the host it runs on, and every node requires exactly one collector. Deployment replicas could be packed onto a subset of nodes, leaving others without coverage.

Why this is the canonical shape for node-level systems. The data being collected lives at the node level: kernel events (syscalls, network packets, file opens), GPU metrics (DCGM via NVML), container runtime events (containerd, CRI-O), node-local resource usage. eBPF programs that read these events run in the kernel of one machine and cannot be shared. Any system whose primary data source is a node must therefore deploy as a DaemonSet: Falco (security via eBPF), Cilium Tetragon (network and process events), DCGM exporter (NVIDIA GPU metrics), node-exporter (host metrics), fluent-bit (container logs), kube-proxy itself, and most CNI agents. The DaemonSet pattern guarantees: one agent per node, automatic enrollment of new nodes, automatic removal on node drain, and access to host paths (/proc, /sys, /var/log, kernel interfaces) via privileged hostPath mounts.

7.4 Streaming data pipeline (logs, security, metrics)

Logs, metrics, and security events share the same pipeline structure: many sources fan in to a buffer, a single processing layer transforms, and the result fans out to multiple sinks.

Many sources fan into a single durable buffer (Kafka). One processing layer reads from the buffer and writes to many sinks. The buffer is the single decoupling point that lets producers and consumers scale, fail, and restart independently.

Key design choices:

  • Local buffering at the agent. If Kafka is briefly unavailable, the agent writes events to local disk so none are lost.
  • Kafka as the decoupling point. Producers and consumers scale independently. Slow processing does not apply backpressure to ingestion.
  • Consumer groups for horizontal scale. The partition count caps consumer parallelism. Provision partitions at peak load × 3.

Why this architecture is the standard for streaming data. Five properties of Kafka-as-buffer make this pattern work where direct producer-to-storage pipelines fail:

  • Speed-mismatch absorption. Sources produce in bursts (a node under attack emits thousands of events per second), but downstream processing has steady throughput. The buffer holds the spike. Without it, either ingestion drops events or processing falls over.
  • Failure isolation. A failing sink (Elasticsearch full, S3 throttled) does not stop ingestion. Events keep flowing into Kafka, and the broken consumer catches up later. In a directly-coupled pipeline, one failed sink takes the whole flow down.
  • Replayability. Kafka retains events for a configurable window (often 7 days). A new analytics consumer added next month can rewind to the start of that window and process historical data without re-instrumenting sources. A failed consumer can rewind and re-process after a fix.
  • Independent scaling per tier. Source agents scale with node count (DaemonSet). Kafka scales by partitions and brokers. Processing scales by consumer-group instances (Flink TaskManagers, Spark executors). Each tier has its own metric and its own autoscaler.
  • Many sinks from one source. The same Kafka topic can feed multiple independent consumer groups: long-term archive (S3), real-time search (Elasticsearch), real-time alerting (SOAR), ML feature pipelines. Adding a new use case is "subscribe to the topic," not "modify the producer."

The design is the same whether the events are container logs, GPU metrics, kernel security events, or application events: fan-in to a durable buffer, single processing layer, fan-out to many sinks. This is also why the pattern composes: the GPU autoscaler's request queue (§7.2) is the same idea applied to inference requests rather than log lines.

Part 8 · The Sharp Edges

Resources, QoS, observability, cost

The operational details that determine whether the cluster runs reliably and cheaply.

8.1 CPU throttling

A common diagnostic pattern: a service shows 30% average CPU but high tail latency. Inspecting the cgroup reveals throttled_time rising rapidly. The CPU is not "busy"; it is being held back by the kernel's CFS scheduler because the Pod has a CPU limit and the kernel is enforcing it per period.

CFS (Completely Fair Scheduler) quota is enforced per period, not as a continuous rate. The default period is 100 ms. A CPU limit of 500m grants the cgroup at most 50 ms of CPU time per 100 ms window. The workload pattern matters as much as the limit:

Limit: 500m  (50 ms / 100 ms period)

Workload A: steady 50 ms of work each period
  ──────────────────────────────────────────
  Period 1: [50 ms work][50 ms idle]   no throttling
  Period 2: [50 ms work][50 ms idle]   no throttling
  Average CPU: 50%   Latency: stable

Workload B: bursty — 4 cores for 12.5 ms, then idle
  ──────────────────────────────────────────
  Period 1: [4 cores × 12.5 ms = 50 ms quota burned]
            [throttled for the remaining 87.5 ms]
  Average CPU: ~12.5%   Latency: SPIKES at every burst

Workload B looks "underutilized" by every average metric, but every request that hits during the throttled window waits up to 87.5 ms for CPU. The Java GC running for 10 ms, a JIT compilation, a sudden batch of incoming requests, all produce this pattern.

# Detect throttling
kubectl exec my-pod -- cat /sys/fs/cgroup/cpu.stat
# nr_periods:    192034
# nr_throttled:  18421       ← ~10% of periods were throttled
# throttled_usec: 95678000   ← ~96 seconds of stalled time

# In Prometheus
rate(container_cpu_cfs_throttled_seconds_total[5m]) > 0.1

Mitigations, in order of preference:

  1. Remove the CPU limit; keep only the request. The process can burst when capacity is available, and the request guarantees scheduler weight under contention. This is the recommended pattern for latency-sensitive services. The trade-off is reduced isolation (a noisy Pod can briefly affect neighbors), which is usually preferable to guaranteed throttling.
  2. Raise the limit-to-request ratio. A limit of 4× the request gives bursty workloads room without removing the cap entirely.
  3. Shorten the CFS period. Setting the kubelet flag --cpu-cfs-quota-period=10ms makes throttling decisions ten times more granular, reducing tail latency at the cost of higher kernel overhead.
  4. Use static CPU manager. For workloads pinned to specific cores (high-frequency trading, real-time inference), CPU manager static policy gives exclusive cores to Guaranteed Pods, bypassing CFS throttling for those cores entirely.

8.2 QoS classes and eviction order

Every Pod is automatically assigned a QoS class based on its resource configuration. Under node-level memory pressure, kubelet evicts Pods in QoS order:

Guaranteedrequest == limit (CPU and memory both set and equal)evicted last
Burstablerequests set, limits > requests or absentevicted next
BestEffortno resources set at allevicted first

The three classes correspond to three concrete YAML configurations:

# Guaranteed: request == limit on both CPU and memory
resources:
  requests: { cpu: "1",   memory: "1Gi" }
  limits:   { cpu: "1",   memory: "1Gi" }

# Burstable: requests set, limits higher (or absent)
resources:
  requests: { cpu: "500m", memory: "512Mi" }
  limits:   { cpu: "2",    memory: "1Gi"   }

# BestEffort: no resources block at all
# (just don't declare requests or limits)

Beyond eviction, QoS class influences three more behaviors:

  • OOM kill priority. When the kernel's OOM killer fires, it ranks processes by oom_score_adj. BestEffort Pods get the highest (most-killable) score; Guaranteed get the lowest.
  • CPU manager static policy. Only Guaranteed Pods with integer CPU requests are eligible for exclusive core pinning.
  • HugePages and topology hints. Many advanced resource features require Guaranteed QoS.

Operational rule: critical services (databases, control-plane components, latency-sensitive APIs) should always be Guaranteed. For extra protection during cluster-level disruption, combine Guaranteed QoS with a PriorityClass:

apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata: { name: critical-service }
value: 1000000
globalDefault: false
description: "Critical workloads, evicted only as last resort"
# Reference in the Pod spec:
# spec.priorityClassName: critical-service

8.3 Observability: logs, metrics, traces

The three pillars of observability answer different questions. Logs report what happened. Metrics report whether the system is trending better or worse. Traces report which hop in a multi-service request is slow. A diagnostic that uses only one pillar will miss most problems.

Logs

Each Pod's stdout / stderr is captured by the container runtime and stored on the node (typically /var/log/pods). A DaemonSet agent (fluent-bit, fluentd, vector) reads these files and ships them to a central store (Loki, Elasticsearch, S3 + Athena). Structured logs (JSON) are mandatory at scale; unstructured logs are unsearchable.

# fluent-bit DaemonSet, mounting host log paths
volumeMounts:
- { name: varlog,  mountPath: /var/log }
- { name: varlibdockercontainers, mountPath: /var/lib/docker/containers, readOnly: true }

Metrics

Prometheus pulls metrics from each Pod's /metrics endpoint at a regular interval (typically 15 s). Pods declare scrape targets through annotations or a ServiceMonitor CRD when running the Prometheus Operator.

# ServiceMonitor: tells Prometheus to scrape this Service
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata: { name: web-api }
spec:
  selector:
    matchLabels: { app: web-api }
  endpoints:
  - port: metrics
    interval: 15s
    path: /metrics

Key Kubernetes metrics worth alerting on:

  • kube_pod_status_phase{phase="Pending"}: how many Pods cannot be scheduled.
  • rate(container_cpu_cfs_throttled_seconds_total[5m]): CPU throttling rate per container.
  • container_memory_working_set_bytes / container_spec_memory_limit_bytes: ratio of actual memory to limit; OOM-kill predictor.
  • kube_pod_container_status_restarts_total: container restart count; rising rate signals a crash loop.
  • node_filesystem_avail_bytes / node_filesystem_size_bytes: disk pressure at the node level.

Traces

Traces show how a single request propagates through multiple services. Each hop creates a span, all spans share a trace ID, and the visualization is a flame graph showing which span took the longest. OpenTelemetry is the standard SDK and protocol.

The hard part is context propagation. The HTTP header traceparent must be forwarded by every intermediate service; one service that drops it breaks the trace from that point onward. Service meshes (Istio, Linkerd) often handle this automatically by injecting sidecars that propagate trace headers across HTTP and gRPC calls.

# Minimal OpenTelemetry Collector DaemonSet config
receivers:
  otlp:
    protocols: { grpc: {}, http: {} }
exporters:
  otlphttp:
    endpoint: https://tempo.observability:4318
service:
  pipelines:
    traces:
      receivers:  [otlp]
      exporters:  [otlphttp]

Diagnosing a real incident with all three

A user reports the API is slow. Diagnostic order:

  1. Metrics first: is the API's http_request_duration_seconds p99 elevated? Yes, it is double normal.
  2. Traces next: pull a slow trace. The flame graph shows 90% of the request time is spent in a downstream service call to user-service.
  3. Metrics on user-service: CPU is normal, but container_cpu_cfs_throttled_seconds_total is rising. Throttling.
  4. Logs on user-service: confirm normal request handling (no errors), only slow.

Conclusion: user-service has a CPU limit too low for its current load, throttling itself, and propagating latency upstream. Fix: raise the CPU limit or remove it. None of the three pillars alone would have surfaced the full picture.

Closing: the unifying model

Kubernetes is a state-reconciliation system composed of multiple independent control loops. Each controller continuously diffs desired state (spec) against actual state (status) and writes a change to close the gap. Pod failures trigger restarts. Node failures trigger rescheduling. Traffic increases trigger HPA scaling. All self-healing properties derive from this single mechanism.

The diagnostic procedure for any production issue is three questions:

① What is the desired state? (How many replicas were declared? Which image? Which probes?)
② What is the actual state? (What is running? Are probes passing? Are endpoints populated?)
③ Which controller is responsible for reconciling this gap, and why is it not acting?

The third question is the diagnostic primitive that distinguishes operational understanding from surface usage.
